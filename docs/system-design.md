## 1. 設計の前提

### MVPのゴール

「写真をアップしてコメントを付けてExcelを出す」この流れが既存の方法より明らかに速いことを検証する。

### 設計の優先順位

```
1. 提出業務の削減（現場監督の実務検証）
2. 写真番号・順序の一貫性（台帳としての信頼性）
3. 将来のAI拡張への対応（再設計コストを最小化）
4. 写真管理としての完成度（Phase 2以降）
```

## 2. 写真番号管理の設計

### 2.1 基本方針：番号は「識別子」ではなく「位置」

工事写真台帳における写真番号は、写真そのものに付いた固定IDではなく、
**出力時の表示順位置**を表す。

```
❌ 悪い設計：写真に番号を固定で持たせる
  → 削除・並び替えのたびに全写真の番号を更新する必要が生じる
  → 並び替えと番号管理が複雑に絡み合う

✅ 正しい設計：番号は表示順から動的に算出する
  → sort_order（並び順）だけを管理する
  → 番号は「sort_orderで並べたときの何番目か」として常に計算する
  → 削除されても自動的に詰まる。更新処理が不要
```

### 2.2 sort_order の実装方針

`sort_order` は連続した整数でなくてよい。大小関係だけが意味を持つ。

| 操作 | sort_order の変化 |
|---|---|
| 初回アップロード | アップロード順に 1000, 2000, 3000... と付与（間隔を持たせる） |
| 写真を1つ上に移動 | 前の写真と前々の写真の中間値を設定（例: 1000と2000の間 → 1500） |
| 写真を1つ下に移動 | 後ろの写真と後々の写真の中間値を設定 |
| 写真を削除 | sort_order の変更不要。欠番になっても表示順の連番は自動で詰まる |
| 大量アップロード後の整合 | sort_order の値が衝突した場合のみ全件を 1000 刻みで振り直す |

この方式（フラクショナルインデックス）により、並び替えのたびに全レコードを更新する処理が不要になる。

### 2.3 アプリ上での写真番号表示

```
表示番号 = sort_order で並べたときの連番（1始まり）
```

- グリッド表示：各写真カードの右上に小さく番号を表示
- 写真詳細画面：「12 / 38」のように現在位置/総数を表示
- 番号はリアルタイムで再計算。削除した瞬間に後続の番号が繰り上がる

### 2.4 Excel出力時の連番

出力ボタンを押した瞬間に、sort_order で並べた写真を 1 から採番する。

```
出力時の採番ロジック（概念）

sort_order でソートした photos の配列を取得
→ 配列のインデックス + 1 を写真番号とする
→ この番号をExcelの写真キャプションに埋め込む

結果：削除・並び替えが何度行われていても、
      出力時点では必ず 1 から始まる欠番のない連番になる
```

## 3. 写真順序変更のUX設計

### 3.1 MVP での実装方針

ドラッグ＆ドロップはモバイルブラウザでの実装難易度が高いため、MVPでは採用しない。

**採用：上下矢印ボタン方式**

```
┌────────────────────────┐
│ [写真] [▲] [▼]  コメント... │
│ [写真] [▲] [▼]  コメント... │
│ [写真] [▲] [▼]  コメント... │
└────────────────────────┘
```

- グリッドビューに「並び替えモード」ボタンを追加
- 並び替えモード中は▲▼ボタンを表示
- ▲：前の写真と sort_order を入れ替え
- ▼：後ろの写真と sort_order を入れ替え
- 先頭の▲・末尾の▼は非表示

**Phase 2 で追加：ドラッグ＆ドロップ**

`@dnd-kit/core` を使用。PC・タブレットでの操作性を大幅に改善できる。

### 3.2 一括並び替えオプション（MVP後）

| 並び順 | 用途 |
|---|---|
| 手動（デフォルト） | 台帳の構成を自分で決めたい場合 |
| 撮影日時順 | EXIFの撮影日時で自動ソート |
| 施工フェーズ順 | 施工前→施工中→施工後の順に並べる |

## 4. Excel出力の設計

### 4.1 MVP テンプレート仕様（固定レイアウト）

**表紙シート**

```
┌─────────────────────────────┐
│        工事写真台帳          │
│                             │
│ 工事名：◯◯工事              │
│ 出力日：2026-05-31           │
│ 写真枚数：24枚               │
└─────────────────────────────┘
```

**写真シート**

```
┌──────────────┬──────────────┐
│   No.1       │   No.2       │  ← 写真番号
│   [写真]     │   [写真]     │  ← セルに写真を埋め込み
│   施工前     │   施工後     │  ← 施工フェーズタグ
│   2026-05-30 │   2026-05-31 │  ← 撮影日時
│   コメント   │   コメント   │  ← コメント（折り返しあり）
├──────────────┼──────────────┤
│   No.3       │   No.4       │
│   ...        │   ...        │
```

- 2列固定レイアウト
- 写真サイズ：セルに合わせて縦横比を維持してリサイズ
- 1ページあたり写真4枚（2列×2行）を基準とする

**MVP で実装しない（Phase 2）**

- 削除時の自動行詰め（出力前に削除すれば出力結果は綺麗なため）
- テンプレートのカスタマイズ
- 目次シート
- 工種・工程・場所タグの列

### 4.2 連番管理の保証

Excel出力において番号の欠番・重複が発生しないことを保証する設計。

```
保証の仕組み

写真の削除  → DBから物理削除
             sort_order は変更しない
             表示番号は取得時に再計算される

並び替え    → sort_order の値を更新
             表示番号は取得時に再計算される

Excel生成   → sort_order ASC でSELECT
             配列のインデックス+1 を採番
             → 必ず1始まりの欠番なし連番が保証される
```

### 4.3 出力フロー

```
[出力ボタン]
  ↓
プロジェクトの写真を sort_order ASC で全件取得
  ↓
各写真の Storage URL から画像データを取得
  ↓
exceljs でブックを生成
  ├── 表紙シートを作成
  └── 写真シートを作成（連番採番・写真埋め込み・コメント挿入）
  ↓
ブラウザで .xlsx ファイルをダウンロード
```

サーバー不要。クライアントサイドで完結する。

## 5. AI拡張性の設計

### 5.1 将来追加する機能のイメージ

```
写真をアップロード or タグを設定した瞬間に

「コンクリート打設後の全景写真です。表面の状態を確認してください。」

というコメント候補を自動生成して入力欄に提示する。
ユーザーは受け入れ・修正・却下を選択できる。
```

### 5.2 AI が必要とするコンテキスト

コメント生成の精度を上げるために必要な情報：

| データ | 取得元 | 用途 |
|---|---|---|
| 写真画像 | Storage URL | 画像認識で被写体を判定 |
| 工事名・現場住所 | projects テーブル | 工事の種類を推定 |
| 施工フェーズタグ | photo_tags | 施工前/中/後を文章に反映 |
| 工種・工程タグ | photo_tags | 専門用語の選定 |
| 写真番号・総枚数 | sort_order から計算 | 「全景」「部分」等の文脈推定 |
| 同プロジェクトの既存コメント | photos テーブル | 文体・表現を統一するため |

**重要：これらの情報はすでに要件定義のデータモデルに存在している。**
AI機能のためにデータ構造を変更する必要はない。

### 5.3 データモデルへの追加（最小限）

photosテーブルに2カラムを追加するだけでAI対応が完了する。

| カラム | 型 | 説明 |
|---|---|---|
| `ai_comment_suggestion` | text, nullable | AIが生成したコメント候補 |
| `ai_suggestion_generated_at` | timestamptz, nullable | 候補を生成した日時 |

- `comment`（人間が書いたコメント）とは完全に分離する
- ユーザーが候補を採用したら `comment` に代入。`ai_comment_suggestion` は変更しない
- 採用率の分析に使える（将来のモデル改善のため）

### 5.4 処理の置き場所

| フェーズ | 実装場所 | 理由 |
|---|---|---|
| MVP | なし | AI機能は未実装 |
| Phase 3（追加時） | Supabase Edge Function | APIキーをサーバー側に秘匿できる。クライアントに露出しない |
| 将来（大規模化後） | 専用APIサーバー | 複数モデルの切り替え・キャッシュ・レート制限の管理が必要になるため |

### 5.5 AIとの通信インターフェース（将来設計）

Edge Functionに以下のインターフェースを定義しておく。
MVP時点では実装しないが、この形に合わせてフロントのコードを書いておく。

```
リクエスト（フロントから）
  photo_id
  project_id
  ※ 他の情報はサーバー側でDBから取得する（クライアントに持たせない）

レスポンス
  suggestion: string    コメント候補文
  confidence: number    信頼度（0〜1）
  reasoning: string     なぜこのコメントにしたか（デバッグ用）
```

フロント側にはコメント入力欄の隣に「AI候補を生成」ボタンを置くUIの枠だけ作っておき、
MVP時点ではボタンを非表示にする。ボタンの表示/非表示を feature flag で制御することで、
コードを変えずにAI機能をON/OFFできる。

### 5.6 モデル選定の考え方（将来）

| モデル | 用途 | 特徴 |
|---|---|---|
| Claude claude-sonnet-4-6（Vision） | コメント生成 | 画像理解 + 日本語の精度が高い |
| Claude claude-haiku-4-5 | 大量処理・低コスト用途 | 速度重視・API単価が低い |

建設専門用語（鉄筋・型枠・打設・養生 等）を含む Few-shot プロンプトを用意することで、
一般的なVision APIより精度の高いコメントを生成できる。

## 7. 技術的判断のまとめ

| 判断 | 採用 | 理由 |
|---|---|---|
| 写真番号の管理方法 | 動的算出（sort_orderから計算） | 削除・並び替えのたびに全件更新する必要がなくなる |
| sort_orderの初期値間隔 | 1000刻み | 挿入・並び替えで中間値を使えるようにするため |
| 並び替えUI（MVP） | 上下矢印ボタン | モバイルブラウザでのDnD実装コストを回避 |
| Excel生成の場所 | クライアントサイド（exceljs） | サーバー不要。Supabase Edge Functionの費用が発生しない |
| AIコメントの保存方法 | 本文と分離（別カラム） | 採用率の計測・モデル改善のフィードバックループを作れる |
| AI処理の場所 | Edge Function（将来） | APIキーの秘匿。クライアントに漏れない |
| AI機能のON/OFF | feature flag | コードを変えずに段階的リリースができる |

## 8. デバイス対応設計

最終更新: 2026-06-03

### 8.1 設計方針

同一URLでスマホとPCの両方から使えるレスポンシブWebアプリとする。
URLもアカウントも共通。デバイス検出はCSS breakpointのみで行い、専用URLへのリダイレクトは行わない。

```
✅ 採用：CSS breakpoint によるレスポンシブ切替
  → 同一コンポーネントがデバイスに応じてレイアウトを変える
  → URLが共通なので、現場でスマホで開いたURLをそのままPCで開ける

❌ 不採用：/mobile / /desktop への振り分け
  → URLが変わると共有・引き継ぎが面倒になる
```

---

### 8.2 ナビゲーションコンポーネントの切り替え

```
src/components/layout/
  AppLayout.tsx        // sm: BottomNav + slim Header / lg: Sidebar + full Header
  Header.tsx           // sm: アプリ名のみ / lg: 工事名・ビュー切替・Excel出力ボタン
  Sidebar.tsx          // lg以上のみ表示
  BottomNav.tsx        // sm以下のみ表示
```

**BottomNav（スマホ）のタブ構成**

| タブ | アイコン | 遷移先 |
|---|---|---|
| ホーム | 🏠 | / （ダッシュボード） |
| 追加 | 📷 | アップロードモーダルを直接開く |
| テンプレ | 📋 | /settings/templates |
| アカウント | 👤 | ユーザーメニュー |

- 「追加」タブはプロジェクト詳細画面にいるときはアップロードモーダルを起動
- ダッシュボードにいるときはプロジェクト新規作成モーダルを起動
- 現在のコンテキストに応じて動作が変わることをアイコンの変化で示す

### 8.3 プロジェクト詳細画面のデフォルトビュー

```
スマホ（〜767px）：グリッドビューをデフォルト
  理由：現場でまず写真の確認・フェーズ設定を行うユースケースに合わせる
  台帳ビューも選択可能（ビュー切替ボタンで）

PC（768px〜）：台帳ビューをデフォルト
```

### 8.4 台帳ビューのレイアウト変化

**スマホ（簡略2行レイアウト）**

```
┌────┬─────────────────────────────────────────┐
│ 01 │ [サムネ 60px]  施工前  05/30  [▲][▼]    │  行1
│    │ [コメント入力欄____________________][T]  │  行2
└────┴─────────────────────────────────────────┘
```

- サムネイルは60px角（スマホで確認できる最小サイズ）
- フェーズ・日時・▲▼を1行に収める
- [T]テンプレートボタンのみ表示（削除は長押しメニュー）

**PC（フルテーブルレイアウト）**

```
┌────┬──────────┬──────────┬──────────────┬─────────────────────┬──────────┐
│ No │ 写真     │ フェーズ │ 撮影日時     │ コメント            │ 操作     │
├────┼──────────┼──────────┼──────────────┼─────────────────────┼──────────┤
│ 01 │[80pxサム]│ 施工前   │ 05/30 07:45  │[入力欄 幅広]        │▲▼ T ×   │
└────┴──────────┴──────────┴──────────────┴─────────────────────┴──────────┘
```

- `<table>` ではなく `<div>` + CSS Grid で実装（レスポンシブ対応を容易にする）
- PCではコメント入力欄の横幅が台帳幅の約50%を占める（コメント入力効率向上）
- Tab / Shift+Tab キーでコメント欄を順番にフォーカス移動できる

### 8.8 コンポーネント実装方針

デバイス分岐はコンポーネント内のCSSクラスで行う。
デバイスごとに別コンポーネントを作ることは避ける（重複実装・メンテコスト増のため）。

```tsx
// 例：台帳行のフェーズバッジ表示
<div className="hidden lg:block">  // PC のみ表示
  <span>撮影日時カラム</span>
</div>
<div className="lg:hidden">        // スマホのみ表示
  <span>省略版日時</span>
</div>
```

**例外：ナビゲーション**
`Sidebar.tsx` と `BottomNav.tsx` は別コンポーネントとして実装する。
これらはレイアウト全体に影響するため、一体化するとコードが複雑になりすぎる。

### 8.9 技術的判断の追加

| 判断 | 採用 | 理由 |
|---|---|---|
| デフォルトビューの差分 | スマホ:グリッド / PC:台帳 | デバイス別の主ユースケースに合わせる |
| ナビゲーション構造 | スマホ:BottomNav / PC:Sidebar | 各デバイスのUI慣習に従う |
| テンプレートUI | スマホ:BottomSheet / PC:インラインパネル | スマホはタッチ操作、PCはキーボード操作に最適化 |
| デバイス検出 | CSS breakpointのみ | URLを変えないことでURLの共有・引き継ぎを容易にする |
| コンポーネント分割 | ナビのみ別コンポーネント、他はCSSで分岐 | 重複実装を最小化しつつナビの複雑さを回避 |
| スマホのExcel出力 | 3点メニュー経由 | ヘッダーを圧迫しない。現場での緊急共有ユースケースはカバー |
